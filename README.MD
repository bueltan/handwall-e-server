

# UDP to xAI Voice Agent Bridge

## Overview

This module acts as a bridge between:

1. A **UDP audio source** (for example, an ESP32 sending PCM audio packets).
2. The **xAI Realtime API** over WebSocket.

Its main responsibilities are:

- Receive audio packets over UDP.
- Reorder packets using sequence numbers.
- Absorb network jitter with a jitter buffer.
- Replace missing packets with silence when packet loss is detected.
- Forward ordered PCM audio to the xAI realtime session.
- Send `commit` events when the input turn is finished.
- Receive audio responses from xAI and store them into a WAV file.
- Log the entire session for debugging and analysis.

---

## High-Level Architecture

The bridge is built around one main class:

- `UdpToXaiBridge`

It runs three async tasks in parallel:

1. **`websocket_sender()`**
   - Sends audio chunks and commit commands to xAI.

2. **`websocket_receiver()`**
   - Receives events from xAI, including transcriptions and audio output.

3. **`udp_loop()`**
   - Receives UDP packets from the device, reorders them, and feeds audio into the async pipeline.

These tasks run together using `asyncio.gather()`.

---

## Main Data Flow

### Input path
UDP device → UDP socket → packet validation → jitter buffer → ordered PCM queue → WebSocket → xAI

### Output path
xAI WebSocket → output audio events → PCM decode → WAV output file

## Configuration

### UDP configuration

- `SERVER_PORT = 5000`  
  UDP port where the bridge listens.

- `BUFFER_SIZE = 4096`  
  Maximum size used by `recvfrom()`.

### Audio format

- `input_sample_rate = 16000`  
  Audio sample rate in Hz.

- `SAMPLES_PER_PACKET = 160`  
  Number of samples per UDP audio packet.

- `PACKET_AUDIO_SIZE = 320`  
  Each sample is 16-bit PCM mono, so `160 × 2 = 320 bytes`.

- `METADATA_SIZE = 4`  
  The first 4 bytes of every UDP packet contain the packet sequence number.

- `PACKET_DURATION_MS = 10 ms`  
  Because 160 samples at 16 kHz represent 10 milliseconds of audio.

### Jitter / packet loss handling

- `JITTER_MS = 250`  
  Target jitter buffer size in milliseconds.

- `JITTER_SIZE`  
  Number of packets represented by the jitter window.

- `MAX_BUFFER = 800`  
  Maximum allowed number of buffered out-of-order packets.

- `MAX_MISSING_BEFORE_LOSS = 5`  
  After several failed attempts to receive the expected packet, the bridge inserts silence and moves forward.

### xAI configuration

- `REALTIME_URL = "wss://api.x.ai/v1/realtime"`
- `VOICE = "eve"`

The session is configured to:
- use PCM input/output audio,
- use transcription with `grok-2-audio`,
- disable automatic turn detection,
- allow tools like `web_search` and `x_search`.

```python
async def connect_xai(self):
```

This method:

1. Reads the API key from `XAI_API_KEY`.
2. Opens a WebSocket connection to the xAI realtime endpoint.
3. Sends a `session.update` event with:

   * selected voice,
   * assistant instructions,
   * tools,
   * transcription model,
   * input audio format,
   * output audio format.

### Important detail

Turn detection is explicitly disabled:

```python
"turn_detection": None
```

That means the bridge must manually decide when user input is finished and explicitly send:

* `input_audio_buffer.commit`
* `response.create`

This is why the UDP side sends a `COMMIT` command.

## `websocket_sender()`

```python
async def websocket_sender(self):
```

This coroutine is responsible for sending outgoing messages to xAI.

It waits on two queues in parallel:

* `audio_queue`
* `commit_queue`

### Behavior

#### If audio is received

It sends:

```json
{
  "type": "input_audio_buffer.append",
  "audio": "<base64 pcm>"
}
```

#### If a commit command is received

It sends:

1. `input_audio_buffer.commit`
2. `response.create`

This tells xAI:

* the user's turn is complete,
* generate the assistant response now.

### Why two queues?

Because audio streaming and turn control are two different event flows:

* audio packets are continuous,
* commit/cancel commands are control signals.

## `websocket_receiver()`

```python
async def websocket_receiver(self):
```

This coroutine listens for events coming back from xAI.

### Handled event types

#### `session.created`

Logs the created session ID.

#### `session.updated`

Confirms the session configuration was applied.

#### `input_audio_buffer.committed`

Confirms xAI accepted the commit.

#### `conversation.item.input_audio_transcription.completed`

Logs the final transcription of the user's audio.

#### `response.output_audio.delta`

Receives base64 PCM chunks from xAI, decodes them, and writes them to the WAV file.

#### `response.output_audio_transcript.delta`

Prints transcript text incrementally to stdout.

#### `response.output_audio.done`

Signals that the assistant audio response is complete.

#### `response.done`

Logs usage statistics such as token counts.

#### `error`

Logs the full error payload from xAI.

### Output audio handling

This module stores assistant output in a WAV file, but it does **not** send that audio back over UDP. It only saves it locally.

## `process_ordered_audio()`

```python
def process_ordered_audio(self):
```

This is one of the most important methods in the module.

Its job is to take out-of-order UDP packets from `self.buffer` and emit them in the correct order.

### Step 1: Send consecutive packets

As long as `expected_seq` exists in the buffer:

* remove the packet,
* push its PCM payload into `audio_queue`,
* increment `expected_seq`.

### Step 2: Detect loss

If the expected packet does not arrive, but there are later packets in the buffer:

* increase `missing_counter`.

When `missing_counter >= MAX_MISSING_BEFORE_LOSS`:

* generate a silence packet,
* send silence into `audio_queue`,
* increment `expected_seq`,
* count it as packet loss.

### Step 3: Trim very old buffered packets

If the reordering buffer grows too much:

* remove packets that are far behind the current sequence.

### Why this matters

UDP does not guarantee:

* delivery,
* order,
* reliability.

This method makes the audio stream robust enough for real-time voice input.

---

## `close_input_turn()`

```python
def close_input_turn(self):
```

This method is called when a UDP `COMMIT` command is received.

### What it does

1. Flushes any remaining in-order packets still waiting in the buffer.
2. Drops leftover out-of-order packets that can no longer be used.
3. Marks the current input turn as closed.
4. Resets:

   * `expected_seq`
   * `missing_counter`
   * packet buffer state

### Why this is needed

When the device decides the user finished speaking, the bridge must:

* finalize the current input audio,
* stop waiting for more UDP packets from that turn,
* allow the commit to be sent cleanly to xAI.

## `udp_loop()`

```python
async def udp_loop(self):
```

This coroutine receives and processes all UDP messages.

### Socket setup

It:

* creates a UDP socket,
* increases the receive buffer size,
* binds to `SERVER_PORT`,
* runs `recvfrom()` through `run_in_executor()` to avoid blocking the event loop.

### Supported UDP control messages

#### `PING`

Replies with `PONG`.

Useful for connectivity testing.

#### `COMMIT`

* logs the event,
* closes the current input turn,
* pushes `"commit"` into `commit_queue`.

#### `CANCEL`

* clears the current buffered input,
* resets sequencing state,
* pushes `"cancel"` into `commit_queue`.

#### `STOP`

* stops the whole bridge.

#### `HELLO_FROM_ESP32...`

* logs the message as informational text.

### Audio packet format

Each audio UDP packet is expected to be:

* first 4 bytes: little-endian sequence number
* remaining 320 bytes: PCM16 mono audio payload

### Validation

The bridge rejects packets when:

* total size is smaller than metadata size,
* audio payload size is not exactly `PACKET_AUDIO_SIZE`.

### Turn reopening logic

If a previous turn was closed and a new packet arrives:

* a new turn is opened automatically,
* `turn_index` is incremented,
* sequencing restarts from the new packet.

### Periodic stats

Every 300 received packets, the bridge logs:

* expected sequence,
* reorder buffer size,
* total received,
* total lost,
* loss rate.

## Turn Model

This module uses a **manual turn-based audio flow**.

### Typical sequence

1. The device sends audio packets over UDP.
2. The bridge buffers and reorders them.
3. The bridge forwards ordered PCM chunks to xAI.
4. The device sends `COMMIT`.
5. The bridge flushes the current input turn.
6. The bridge sends:

   * `input_audio_buffer.commit`
   * `response.create`
7. xAI returns transcription and audio output.
8. Assistant output audio is written to the WAV file.

### Important implication

Without `COMMIT`, xAI will continue waiting for more user audio.

---

## Packet Loss Strategy

The bridge does **not** block forever waiting for missing UDP packets.

Instead:

* if a packet is missing temporarily, it waits a few cycles,
* if the packet still does not arrive, it inserts silence,
* then it continues with the next sequence.

### Benefit

This keeps the real-time stream moving forward and prevents audio from freezing because of one lost packet.

### Tradeoff

Some brief audio content may be replaced by silence.

---

## Strengths of This Design

* Works well with unreliable UDP transport.
* Preserves real-time flow with packet reordering.
* Handles missing packets gracefully.
* Separates control events from audio events.
* Produces logs and WAV output for debugging.
* Keeps the WebSocket event loop responsive by offloading blocking UDP reads.

---

## Current Limitations

1. **Assistant audio is only saved to WAV**

   * It is not streamed back to the UDP device.

2. **No automatic voice activity detection**

   * The sender must explicitly send `COMMIT`.

3. **Hardcoded configuration values**

   * Port, sample rate, voice, and jitter settings are constants in the script.

4. **No reconnection strategy**

   * If the websocket fails, the script logs the error and stops.

5. **API key fallback is hardcoded in code**

   * This should be avoided in production for security reasons.

---

## Recommended Improvements

### 1. Move all comments and log messages to English

Some comments and log text are still in Spanish. For consistency, all of these should be rewritten in English.

### 2. Remove hardcoded API key fallback

Use only:

```python
os.environ["XAI_API_KEY"]
```

or fail clearly if the variable is missing.

### 3. Stream assistant audio back to the device

If needed, `response.output_audio.delta` could be sent back over UDP.

### 4. Add inactivity-based auto-commit

If the sender stops transmitting audio for a defined timeout, the bridge could auto-commit the current turn.

### 5. Make configuration external

Move settings into:

* environment variables,
* a JSON file,
* or command-line arguments.

### 6. Improve cancel behavior

Currently cancel only logs `"CANCEL requested"` in the sender path. It could also send a proper interruption event to xAI if supported by the API.

## Summary

This module is a **real-time UDP audio bridge** that connects an external microphone/audio sender to the xAI realtime voice API.

It solves several practical problems:

* packet ordering,
* jitter buffering,
* packet loss concealment,
* explicit turn finalization,
* output capture,
* structured logging.

In short, it provides the transport and session-control layer needed to connect a simple UDP audio source to a conversational realtime voice model.
